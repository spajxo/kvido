# State CLI Wrappers Design

GitHub issue: #74

## Problem

Instruction markdowns (skills, agents, hooks, commands) reference `state/` files directly. This leaks internal file layout into LLM prompts, complicates validation, and makes future refactors of `state/` risky.

## Goal

Replace direct `state/` file references in instruction markdowns with `kvido` CLI commands. Raw paths remain only in bash implementation scripts.

## Design Decisions

- **Flat namespace** — new commands follow existing pattern (`kvido current`, `kvido planner-state`, `kvido source-health`), consistent with `kvido task`, `kvido heartbeat-state`, `kvido log`.
- **`planner-state` migrates to JSON** — structured data (events, timestamps, reminders) benefits from `jq` + `flock` atomicity. Markdown was a poor fit for dedup tables and key-value pairs.
- **`current` stays markdown** — free-form text, read/written whole. No parsing needed.
- **`session-context` gets no CLI wrapper** — generated by `build-context.sh`, injected via `kvido context session`. Agents receive it implicitly.
- **`plugin-proposals/` deferred** — empty in production, no wrapper needed yet.

## Command Surface

### `kvido current` (new)

| Command | Description |
|---------|-------------|
| `kvido current get` | Print `state/current.md` to stdout |
| `kvido current set` | Read stdin, write to `state/current.md` |

Implementation: `skills/current/current.sh` — trivial cat read/write.

### `kvido planner-state` (new — replaces markdown file)

#### Events (dedup)

| Command | Description |
|---------|-------------|
| `kvido planner-state event check <key>` | Exit 0 if exists, exit 1 if not |
| `kvido planner-state event report <key>` | Upsert: set `first_seen` on create, update `last_reported` always |
| `kvido planner-state event list` | Print all events as JSON to stdout |
| `kvido planner-state event cleanup [--max-age 48h]` | Remove events older than threshold (default 48h) |

#### Timestamps

| Command | Description |
|---------|-------------|
| `kvido planner-state timestamp get <key>` | Print value to stdout, exit 1 if missing |
| `kvido planner-state timestamp set <key> <value>` | Set key to value |

#### Reminders

| Command | Description |
|---------|-------------|
| `kvido planner-state reminder get <slug>` | Print `last_reminded` date, exit 1 if missing |
| `kvido planner-state reminder set <slug>` | Set `last_reminded` to today |

#### Schedule

| Command | Description |
|---------|-------------|
| `kvido planner-state schedule get` | Print schedule text to stdout |
| `kvido planner-state schedule set` | Read stdin, write schedule text |

#### Last Run

| Command | Description |
|---------|-------------|
| `kvido planner-state last-run get` | Print last run metadata as JSON |
| `kvido planner-state last-run set` | Read stdin JSON, write last run metadata |

### `kvido source-health` (new)

| Command | Description |
|---------|-------------|
| `kvido source-health get [<source>]` | Print full JSON or single source status |
| `kvido source-health set <source> <status>` | Update one source entry, set timestamp to now |

### Unchanged

- `kvido task` — already abstracts `state/tasks/`
- `kvido heartbeat-state` — already abstracts `state/heartbeat-state.json`
- `kvido log` — already abstracts `state/log.jsonl`

### `state/current.md` — unchanged

Free-form markdown, read/written whole.

### `state/source-health.json` — unchanged

Already JSON. New CLI wrapper only.

## Rule

After migration, raw `state/` paths appear only in:
- Bash implementation scripts (`skills/*/**.sh`, `hooks/build-context.sh`)
- `CLAUDE.md` (architecture documentation)

Never in `.md` files that serve as LLM instructions (agents, skills, commands, hooks).

## Implementation Order

1. `skills/planner-state/planner-state.sh` — new script, JSON format, full subcommand surface
2. `skills/current/current.sh` — trivial read/write wrapper
3. `skills/source-health/source-health.sh` — jq wrapper
4. Extend `kvido` dispatcher — register `current`, `planner-state`, `source-health`
5. One-time migration script — convert `state/planner-state.md` to `state/planner-state.json`
6. Update all instruction markdowns per migration table
7. Update `commands/setup.md` — validate new files exist
8. Remove `state/planner-state.md` references from codebase

## Migration Script

For existing installations, provide `skills/planner-state/migrate.sh` that:
1. Reads `state/planner-state.md`
2. Parses sections with awk/sed into JSON structure
3. Writes `state/planner-state.json`
4. Renames old file to `state/planner-state.md.bak`

Setup command detects stale `.md` and runs migration automatically.
